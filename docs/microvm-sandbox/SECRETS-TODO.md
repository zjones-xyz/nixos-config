## 1. Guest's sops age identity (blocks everything else below)

The guest needs its own SSH host key, distinct from every fleet host's, to serve as its
sops age identity — see `DECISIONS.md`'s Secrets section for why this must live on the
persistent state volume. As of the Phase 3 code, the guest's `/etc/ssh/ssh_host_ed25519_key`
is auto-generated by `sshd-keygen` on first boot (no manual generation step needed) and
persisted across restarts via `environment.persistence."/persist"` (impermanence) — so
step 1 below is "read it," not "generate it."

1. On first guest boot, read its `/etc/ssh/ssh_host_ed25519_key.pub` (SSH in as `agent`
   once Phase 3 is deployed, or read it via the host if there's still no interactive
   guest console at that point).
2. `ssh-to-age < ssh_host_ed25519_key.pub` (on the guest, or copy the pubkey out and run
   it wherever `ssh-to-age` is available).
3. Replace the placeholder in `.sops.yaml`'s `&agent-sandbox` anchor with the real age
   pubkey from step 2.
4. Create `secrets/agent-sandbox.yaml` (`sops secrets/agent-sandbox.yaml` — sops creates
   it fresh since it doesn't exist yet) with the tokens below. The creation rule already
   in `.sops.yaml` encrypts it to `admin` + `*agent-sandbox`.
5. `sops updatekeys secrets/agent-sandbox.yaml` once the file and the real key exist.
6. `nixos-rebuild switch --flake .#pegasus` again — this is what actually activates the
   `sops = lib.mkIf (builtins.pathExists ...) { ... }` block in
   `modules/nixos/microvm-sandbox.nix`, since it's gated on the file's existence.

## 2. `CLAUDE_CODE_OAUTH_TOKEN`

Minted via the Claude Code CLI's **subscription** flow, not an API key:

```
claude setup-token
```

Run this wherever Zoe is already authenticated to her Claude subscription (e.g. on
Serenity). Store the resulting token as a `CLAUDE_CODE_OAUTH_TOKEN: <value>` key in
`secrets/agent-sandbox.yaml` under sops (`sops secrets/agent-sandbox.yaml` to edit in
place) — the key name must match `sops.secrets."CLAUDE_CODE_OAUTH_TOKEN"` in
`modules/nixos/microvm-sandbox.nix` exactly, since sops-nix decrypts by matching key name.

## 3. Codex CLI token — NOT symmetric with Claude's, needs a decision first

Researched (2026-07-21) rather than assumed, and it turned out genuinely different from
Claude Code's flow — see `DECISIONS.md`'s Phase 3 section for the full account. Summary:

- Codex's subscription-based auth is **file-based** (`~/.codex/auth.json`, a JSON blob
  with access/refresh tokens), not an env-var token like Claude's.
- Codex **rewrites that file in place** as tokens refresh — which doesn't mesh cleanly
  with sops-nix's "decrypt once at activation" model; a future `switch` could silently
  reset it back to a stale value if sops-nix ever redeploys that secret.
- OpenAI's own current guidance actually recommends `OPENAI_API_KEY` (pay-as-you-go, not
  subscription) for automation/CI use — the opposite emphasis from Claude Code's
  subscription-first `setup-token` flow.

**No token wiring implemented for Codex yet** — `pkgs.codex` is installed regardless
(the CLI binary itself), but the actual auth mechanism needs a decision: accept the
sops-nix-reset risk and wire up `auth.json` the same way as Claude's token (mint via
`codex login` on a machine with a browser, e.g. Serenity, then feed the resulting
`~/.codex/auth.json` contents into a new sops secret), fall back to `OPENAI_API_KEY` (a
regular env-var secret, much simpler, but pay-as-you-go rather than subscription), or
treat this as a manual out-of-band step (copy `auth.json` onto the persistent volume
directly, outside of sops/this repo entirely, matching how the LUKS-unlock initrd key is
handled). Not decided as of this writing.
